# 在线聊天系统 — 技术方案

> **技术栈**：前端 HTML + CSS + JS，后端 Spring Boot (MVC)，数据库 MySQL
>
> **文档版本**：v1.0 | **生成日期**：2026-05-28

---

## 一、系统架构总览

### 1.1 分层架构

```
┌─────────────────────────────────────────────────┐
│                  Browser (PC/移动端)              │
│           HTML + CSS + JavaScript                │
└────────────┬──────────────────┬──────────────────┘
             │ HTTP (REST API)  │ WebSocket (STOMP)
             ▼                  ▼
┌─────────────────────────────────────────────────┐
│              Spring Boot (port 8080)             │
│                                                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────────┐   │
│  │Controller│  │WebSocket │  │  Static/     │   │
│  │  Layer   │  │  Handler │  │  (前端资源)    │   │
│  └────┬─────┘  └────┬─────┘  └──────────────┘   │
│       │              │                           │
│  ┌────▼──────────────▼──────┐                    │
│  │      Service Layer       │                    │
│  └────────────┬─────────────┘                    │
│               │                                  │
│  ┌────────────▼─────────────┐  ┌──────────────┐  │
│  │   MyBatis-Plus (Mapper)  │  │  File Storage │  │
│  └────────────┬─────────────┘  │  (uploads/)   │  │
│               │                └──────────────┘  │
└───────────────┼──────────────────────────────────┘
                ▼
       ┌──────────────┐
       │   MySQL 8.0   │
       └──────────────┘
```

### 1.2 核心通信模型

- **请求/响应类操作**（注册、登录、好友申请、群管理、记录查询、导出）→ REST API
- **实时推送类操作**（聊天消息、在线状态变更、好友申请通知）→ STOMP over WebSocket
- **统一模式**：发消息时写 DB + WebSocket 推送；上线时根据时间戳拉取未读消息

### 1.3 统一响应格式

```json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
```

### 1.4 WebSocket 消息格式

```json
{
  "type": "CHAT / FRIEND_REQUEST / STATUS_CHANGE",
  "from": 1,
  "to": 2,
  "groupId": null,
  "content": "...",
  "messageType": "TEXT / IMAGE / FILE / VOICE",
  "timestamp": "2026-05-28 12:00:00"
}
```

### 1.5 关键技术决策

| 决策项 | 选型 | 理由 |
|--------|------|------|
| 前端部署 | Spring Boot `static/` 目录 | 同域同端口，无需 CORS，部署简单 |
| 实时通信 | STOMP over WebSocket | Spring 原生支持，轻量够用 |
| 认证方式 | JWT Token | 无状态，前端存 localStorage，WebSocket 握手也用 Token |
| ORM 框架 | MyBatis-Plus | CRUD 开箱即用，分页插件方便 |
| 语音方案 | 仅语音消息（MediaRecorder + 文件上传） | 不做实时语音通话，降低复杂度 |
| 文件存储 | 本地磁盘 `uploads/` 目录 | 课设场景无需 OSS |
| 离线消息 | 全部持久化到 DB，上线后拉取 | 逻辑统一，实现简单 |

## 二、各成员职责与开发内容

### 2.1 成员 A — 用户与认证（公共基础负责人）

**核心定位**：第一个开工的人，搭脚手架 + 认证体系 + 部署容器化，其他人依赖 TA 的产出。

#### 后端接口清单

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 注册 | POST | `/api/auth/register` | 用户名唯一校验，密码 BCrypt 加密，返回用户信息 |
| 登录 | POST | `/api/auth/login` | 校验密码，签发 JWT Token，更新 last_login_time |
| 登出 | POST | `/api/auth/logout` | Token 加入黑名单（Redis 或内存表），更新状态为离线 |
| 获取个人信息 | GET | `/api/user/profile` | 从 Token 解析 userId 查询 |
| 更新个人信息 | PUT | `/api/user/profile` | 修改昵称、头像、签名 |
| 修改密码 | PUT | `/api/user/password` | 验证旧密码 → 更新新密码 |
| 切换在线状态 | PUT | `/api/user/status` | 0-离线 1-在线 2-忙碌 3-勿扰 |
| 上传文件 | POST | `/api/file/upload` | 通用文件上传，返回文件 URL，限制类型和大小 |
| 下载文件 | GET | `/api/file/{filename}` | 文件下载/访问 |

#### 公共基础设施（必须在其他人开发前完成）

- `Result<T>` 统一响应封装类
- `GlobalExceptionHandler` 全局异常处理（`@RestControllerAdvice`）
- `JwtUtil` 工具类（生成 Token、解析 Token、校验过期）
- `AuthInterceptor` 认证拦截器（拦截除 `/api/auth/*` 外的所有请求，解析 Token 注入 userId）
- `WebMvcConfig` 注册拦截器、配置 CORS
- `MyBatisPlusConfig` 分页插件配置、自动填充（create_time、update_time）
- `WebSocketAuthInterceptor` WebSocket 握手拦截器（从 STOMP Header 取 Token 校验）
- `FileUtil` 文件上传工具（生成唯一文件名、校验类型大小、存储到 uploads/）
- 数据库迁移框架：`db/init.sql`（建库）+ `db/migration/*.sql`（各成员各自的建表语句）+ `db/migrate.sh`（自动迁移脚本）

#### 前端页面与组件

- `login.html` — 登录表单（用户名、密码、记住我 checkbox、错误提示）
- `register.html` — 注册表单（用户名、密码、确认密码、昵称、头像上传预览、实时校验）
- `profile.html` — 个人信息编辑（头像裁剪上传、昵称、签名、密码修改弹窗、状态切换下拉）
- `index.html` — **主界面骨架**（这是最重要的公共页面）：
  - 左侧面板：搜索栏 + Tab 切换（好友列表 / 群列表 / 最近会话）
  - 右侧面板：聊天窗口占位区
  - 顶部导航栏：头像、状态切换、退出、管理后台入口（管理员可见）
  - 底部状态栏：连接状态指示

#### 前端公共 JS

- `api.js` — 封装 fetch，统一加 Token Header，统一错误处理
- `auth.js` — Token 存取、登录状态检查、未登录跳转
- `websocket.js` — STOMP 连接管理、自动重连、消息分发（其他模块注册回调）
- `utils.js` — 时间格式化、消息类型判断等通用函数

#### 部署与容器化（新增）

- `Dockerfile` — 多阶段构建（前端静态资源编排 + Spring Boot JAR 打包）
- `docker-compose.yml` — 编排 Spring Boot 应用 + MySQL 数据库
- `deploy.sh` — 一键部署脚本（构建镜像、启动容器、初始化数据库）
- `db/init.sql` — 数据库初始化脚本（建库）
- `db/migration/*.sql` — 各成员各自的建表语句
- `db/migrate.sh` — 自动迁移脚本
- `.dockerignore` — 排除无关文件
- CI/CD 配置（可选，如 GitHub Actions 自动构建）

#### 交付物与验收标准

- 所有 API 用 Postman 可测通
- 前端登录注册流程跑通，Token 鉴权生效
- WebSocket 连接建立成功，握手时 Token 校验通过
- `db/migration/` 目录包含全部建表语句，`migrate.sh` 一键建库可用

### 2.2 成员 B — 私聊与实时通信（WebSocket 基础负责人）

**核心定位**：搭 WebSocket 通信层，群聊和语音都依赖 TA 的基础设施。

#### 后端接口清单

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 发送私聊消息 | POST | `/api/message/private/send` | 写 DB（status=未读）+ STOMP 推送给对方 |
| 获取历史消息 | GET | `/api/message/private/history` | 参数：friendId、pageNum、pageSize，按时间倒序 |
| 获取未读消息 | GET | `/api/message/private/unread` | 根据登录时间戳拉取未读，登录后调用 |
| 标记已读 | PUT | `/api/message/private/read` | 参数：friendId，将该会话所有未读改为已读 |
| 获取最近会话列表 | GET | `/api/message/private/recent` | 返回最近 N 个聊天对象 + 最后一条消息 + 未读数 |

#### WebSocket 管理

- `WebSocketSessionManager` — 维护 `userId → Session` 映射，支持同一用户多端在线
- `ChatMessageHandler` — 处理 `/app/chat.send` 目的地，接收消息 → 持久化 → 推送
- STOMP 目的地设计：
  - 客户端发送：`/app/chat.send`
  - 服务端推送（点对点）：`/user/queue/messages`
  - 在线状态广播：`/topic/status`
- 连接事件：`connect` 时更新用户状态为在线、广播上线通知；`disconnect` 时更新离线、广播下线

#### 前端页面与组件

- `chat-panel.html` — 聊天窗口区域（嵌入 index.html 右侧）：
  - 消息列表区（消息气泡：区分自己/对方、文字/图片/文件/语音不同样式）
  - 每条消息：头像、昵称、内容、时间戳、已读/未读标记
  - 输入区：文本输入框 + 发送按钮 + 工具栏（表情、图片、文件、语音按钮占位）
  - 加载更多按钮（上拉加载历史消息）
- `conversation-list.html` — 最近会话列表（嵌入 index.html 左侧）：
  - 会话项：头像、昵称、最后消息摘要、时间、未读角标
  - 点击会话切换右侧聊天窗口

#### 前端 JS

- `chat.js` — 聊天核心逻辑（发送消息、接收消息、渲染气泡、滚动加载）
- `conversation.js` — 会话列表管理（排序、未读数更新、选中高亮）

#### 协作依赖

- **依赖 A**：复用 `websocket.js` 的连接管理、`api.js`、`index.html` 主界面骨架
- **被 C 依赖**：群消息复用同一个 STOMP 连接，推送到 `/topic/group/{groupId}`
- **消息格式约定**：`{ type, from, to, groupId, content, messageType, timestamp }`

#### 交付物与验收标准

- 两个用户能实时互发文字消息
- 离线用户上线后能拉取到未读消息
- 已读/未读状态正确切换
- 最近会话列表按时间排序，未读角标正确

#### 协作依赖

- **依赖 A**：语音文件上传复用 A 的 `/api/file/upload` 接口；管理后台认证拦截复用 A 的 AuthInterceptor，额外加角色校验

## 三、数据库表设计

### 3.1 实体关系总览

```
user ──< friendship >── user
  │        │
  │        └──> friend_group
  │
  ├──< private_message
  │
  ├──< chat_group (owner)
  │       │
  │       ├──< group_member >── user
  │       │
  │       └──< group_message
  │
  ├──< voice_message (关联 private_message / group_message)
  │
  ├──< file_record
  │
  ├──< announcement (admin 发布)
  │
  └──< sensitive_word
```

### 3.2 需要设计的表

| 表名 | 负责人 | 说明 |
|------|--------|------|
| `user` | A | 用户基础信息、密码、状态、角色 |
| `private_message` | B | 私聊消息，记录发送者、接收者、内容、类型、已读状态 |
| `chat_group` | C | 群聊基础信息（群名、头像、公告、群主） |
| `group_member` | C | 群成员关系（群ID、用户ID、角色、免打扰） |
| `group_message` | C | 群聊消息（群ID、发送者、内容、类型） |
| `friendship` | D | 好友关系（申请人、被申请人、状态、所属分组） |
| `friend_group` | D | 好友分组（分组名、排序、是否默认） |
| `file_record` | E | 文件上传记录（可选，记录上传文件元信息） |
| `announcement` | F | 系统公告（标题、内容、发布者、发布时间） |
| `sensitive_word` | F | 敏感词（词语、替换词） |
| `voice_message` | F | 语音消息扩展（关联消息ID、文件地址、语音时长、文件大小） |

### 3.3 实体关系说明

**核心关系：**

- `user` 与 `user` 通过 `friendship` 建立多对多好友关系，`friend_group` 作为分组维度
- `user` 与 `chat_group` 通过 `group_member` 建立多对多群成员关系
- `private_message` 是 `user` 与 `user` 之间的点对点消息记录
- `group_message` 是 `chat_group` 内的消息记录
- `voice_message` 作为 `private_message` / `group_message` 的扩展，存储语音文件地址和时长

**设计要点：**

- 每张表都需要 `id`（主键自增）、`create_time`、`update_time` 字段
- `user` 表的 `password` 必须 BCrypt 加密存储
- `private_message` 和 `group_message` 的 `message_type` 字段区分文字(0)、图片(1)、文件(2)、语音(3)
- `friendship` 的 `status` 字段：0-待验证、1-已通过、2-已拒绝
- `friendship` 需要双向记录（A→B 和 B→A），接受时同时插入两条
- `group_member` 的 `role` 字段：0-成员、1-管理员、2-群主
- `user` 表增加 `enabled` 字段（0-禁用、1-启用）供管理员使用
- `voice_message` 是消息的扩展表，通过 `message_id` 关联 `private_message` 或 `group_message`，存储语音特有的文件地址和时长
